# 🤖 Smart Vending Machine API

REST API для управления умным торговым автоматом. Проект реализован на **TypeScript + Express.js** с использованием in-memory хранения данных.

---

## 🚀 Стек технологий

* **Язык:** TypeScript
* **Платформа:** Node.js
* **Фреймворк:** Express.js
* **Архитектура:** REST API

---


## 📡 API Эндпоинты

### 1. Получить состояние автомата
* **Метод:** `GET`
* **Путь:** `/machine`
* **Response (200 OK):**
  ```json
  {
    "temperature": 20,
    "credit": 0,
    "revenue": 0,
    "slots": [],
    "status": "OPERATIONAL"
  }
  ```

### 2. Добавить товар (пополнение автомата)
* **Метод:** `POST`
* **Путь:** `/machine/restock`
* **Описание:** Создает новый слот или обновляет существующий. Показатель `freshness` автоматически устанавливается в `100`.
* **Request Body:**
  ```json
  {
    "id": 1,
    "product": "Cola",
    "price": 120,
    "stock": 10
  }
  ```

### 3. Внести деньги
* **Метод:** `POST`
* **Путь:** `/machine/insert`
* **Описание:** Увеличивает текущий кредит пользователя на указанную сумму.
* **Request Body:**
  ```json
  {
    "amount": 100
  }
  ```

### 4. Купить товар
* **Метод:** `POST`
* **Путь:** `/machine/select`
* **Описание:** Перед покупкой проверяется наличие слота, наличие товара, его свежесть, достаточность средств на балансе и текущий статус автомата.
* **Request Body:**
  ```json
  {
    "slotId": 1
  }
  ```

### 5. Обслуживание автомата (технические работы)
* **Метод:** `POST`
* **Путь:** `/machine/maintain`
* **Описание:** Снижает температуру автомата на 30 единиц. Операция недоступна, если автомат уже находится в состоянии `BROKEN`.

---

## 🌡 Статусы автомата

Текущее состояние автомата напрямую зависит от его внутренней температуры:

| Температура | Статус | Описание |
| :--- | :--- | :--- |
| **<= 80** | `OPERATIONAL` | Автомат работает исправно. |
| **81 - 100** | `OVERHEATED` | Автомат перегрет, но продолжает функционировать. |
| **> 100** | `BROKEN` | Автомат сломан. Любые операции изменения состояния блокируются. |

---

## ⏱ Автоматические процессы

В системе запущен фоновый планировщик (`scheduler`). Каждую **60 секунд** он выполняет следующие действия:
* Увеличивает температуру автомата на `3` градуса.
* Уменьшает показатель свежести (`freshness`) во всех заполненных слотах на `1`.

> ⚠️ **Важно:** Если в процессе нагрева температура превышает `100` градусов, автомат окончательно переходит в статус `BROKEN` и все дальнейшие изменения параметров автоматически останавливаются.

## ❌ Обработка ошибок

В случае некорректного запроса или невозможности выполнить операцию, API возвращает соответствующий HTTP-статус и тело ошибки.

**Примеры ответов:**
```json
{
  "message": "Slot not found"
}
```
```json
{
  "message": "Machine is broken"
}
```

### Используемые HTTP-статусы:
* `400 Bad Request` — Переданы некорректные данные или не пройдены правила валидации.
* `404 Not Found` — Запрашиваемый ресурс (например, ID слота) не найден.
* `409 Conflict` — Операция невозможна в текущем состоянии автомата (например, покупка при статусе `BROKEN`).

### 🏢 Архитектурный стиль: Слоистый монолит (Layered Monolith)

Приложение построено по принципу классического слоистого монолита (N-Tier Architecture) с разделением ответственности по техническим слоям:

* **Presentation Layer (Маршруты и Контроллеры):** Принимает HTTP-запросы, отвечает за транспорт и формат ответов.
* **Business Logic Layer (Сервисы):** Содержит всю бизнес-логику автомата (расчет температуры, покупку товара, смену статусов).
* **Data Access Layer (In-Memory Storage):** Централизованно хранит и предоставляет состояние автомата в оперативной памяти.

Все слои работают в рамках единого Node.js процесса, что обеспечивает максимальную скорость межслойного взаимодействия и простоту отладки.е расходы:** Отсутствует необходимость в сложной сетевой инфраструктуре или брокерах сообщений для синхронизации состояния.
